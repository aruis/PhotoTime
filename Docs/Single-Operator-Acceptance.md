# Single-Operator Acceptance (v1)

适用场景：当前阶段由 1 位产品负责人（你）完成体验验收，开发侧（我）按结果迭代修复。  
目标：在没有外部用户的前提下，仍然形成可重复、可量化的交付判断。

## 0. 前置条件（进入验收前必须满足）

1. 应用可在目标机器上正常启动（无 Gatekeeper 阻断，或有可接受的绕过方式）。
2. 当前使用 ad-hoc 签名，`spctl --assess` 会被拒绝——在分发给外部用户之前需完成 Developer ID 签名 + Notarization 或明确绕过说明。
3. 前置条件未满足时，验收仍可进行（用于发现产品问题），但结果不可作为"发布就绪"判定。

## 1. 使用方式

1. 每轮迭代后，按下面 10 个任务完整走一遍。
2. 每个任务按同一标准打分并记录阻塞点。
3. 开发仅优先修复 Top 3 高影响问题（不扩散）。

## 2. 任务清单（每轮必测）

1. 空状态启动后，是否能在 30 秒内知道第一步该做什么。
2. 导入 20-30 张真实照片（含横图/竖图）是否顺畅，无明显困惑。
3. 预检（Preflight）问题是否可理解，且能定位到对应素材。
4. 调整核心参数（时长/转场/风格/方向）时，反馈是否清晰。
5. 单帧预览与时间轴预览是否一致、可预期。
6. 导入单轨音频后，预览与导出行为是否符合预期。
7. 导出一次完整视频是否成功，耗时是否可接受（记录实际耗时）。
8. 导出失败时，失败卡片是否清楚回答"问题是什么/下一步做什么"。
9. 修正问题后重试导出是否顺畅，不需要额外猜测操作路径。
10. 成片回看（系统播放器）质量是否达到"可交付"标准。
    - 关注维度：分辨率与设置一致、转场衔接自然无跳帧、音频与画面同步、边距/相框渲染正确、淡入淡出（如开启）正常。

## 3. 评分规则（每项 0-3 分）

- `3`：无阻塞，体验顺滑，可直接交付。
- `2`：可完成但有轻微摩擦，不影响本轮发布。
- `1`：可完成但有明显困惑/绕路，需要优先修复。
- `0`：无法完成或结果错误，属于发布阻塞。

## 4. 通过门槛

1. 任一任务为 `0`：本轮不通过，先修阻塞。
2. `1` 分任务超过 2 个：本轮不通过，继续收敛。
3. 总分至少 `24/30` 且无 `0`：可进入发布准备。
4. 进入发布准备还需同时满足"前置条件"第 1、2 项。
